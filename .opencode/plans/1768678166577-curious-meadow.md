# Plan: Refactor ClientSendGrain frame state locking

## Goal
Replace the contested lock around `_frameStates` in `ClientSendGrain` with a `ConcurrentDictionary` and lock-free state transitions while keeping behavior close to existing (small shifts acceptable).

## Scope
- Only `src/RemoteViewer.Server/Orleans/Grains/ClientSendGrain.cs`.
- Replace `_sync` usage tied to `_frameStates`.
- No other locks or subsystems.

## Design
- Change `_frameStates` to `ConcurrentDictionary<string, FrameSendState>` (StringComparer.Ordinal).
- Make `FrameSendState` concurrency-safe using atomic fields:
  - `int InFlight` (0/1) via `Interlocked.CompareExchange` and `Interlocked.Exchange`.
  - `QueuedMessage? PendingFrame` swapped with `Interlocked.Exchange`.
- Keep async send outside any lock.

## Implementation Steps
1. Update fields in `ClientSendGrain`:
   - Replace `Dictionary` with `ConcurrentDictionary`.
   - Remove `_sync` or keep it only if still required for non-frame lifecycle (ideally remove entirely).
2. Update `EnqueueFrame`:
   - `GetOrAdd` state per connection.
   - If `InFlight` transitions 0->1, send immediately.
   - Otherwise, swap `PendingFrame` atomically, log dropped frame if replaced.
3. Update `AckFrame`:
   - If `PendingFrame` exists, swap it out and send it (keep `InFlight=1`).
   - Else set `InFlight=0`.
   - Optional: remove empty state with `TryRemove` when no in-flight and no pending.
4. Update error handling in `SendFrameAsync`:
   - On failure, atomically clear `PendingFrame` and set `InFlight=0` for that connection.
5. Ensure no awaits within any remaining locks (if any remain).
